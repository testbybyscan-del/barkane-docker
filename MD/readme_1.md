- **`php/Dockerfile`** — использовал `php:8.3-fpm-alpine`, установил `postgresql-dev` и `pdo_pgsql`, скопировал Composer (на всякий случай), установил права www-data.

## 3. Решение проблем с автозагрузкой классов

### 3.1. Ошибка `Class "App\FilterBuilder" not found`

При попытке фильтрации через `events.php` возникала фатальная ошибка, указывающая на отсутствие класса `App\FilterBuilder` в `EventRepository.php`. Проверка показала, что:

- Файл `FilterBuilder.php` лежит в `app/src/` и содержит правильный неймспейс `namespace App;`.
- Однако автозагрузка Composer не была настроена, так как отсутствовал `composer.json`.

### 3.2. Создание `composer.json` и генерация автозагрузки

В папке `app/` я создал `composer.json`:

```json
{
    "autoload": {
        "psr-4": {
            "App\\": "src/"
        }
    }
}
```

Затем внутри контейнера `barkane_php` выполнил:

```bash
php -r "copy('https://getcomposer.org/installer', 'composer-setup.php');"
php composer-setup.php --quiet
php -r "unlink('composer-setup.php');"
php composer.phar install --no-dev --optimize-autoloader
```

Это сгенерировало `vendor/autoload.php` и карту классов.

### 3.3. Подключение автозагрузки в скриптах

В `events.php` и `import.php` добавил строку:

```php
require_once __DIR__ . '/../vendor/autoload.php';
```

Теперь все классы из пространства `App\` загружаются автоматически. Ошибка исчезла.

## 4. Перенос статических файлов

В логах Nginx появились ошибки `404` на `/css/styles.css`, `/js/datepicker.js` и `/images/640360barkane.jpg`. Оказалось, что в папке `app/` эти файлы лежат в корне, а HTML-шаблоны ссылаются на подпапки.

Я создал необходимые каталоги и переместил файлы:

```bash
cd ~/dev/docker-containers/barkane-docker/app
mkdir css js images
mv styles.css css/
mv datepicker.js check.js filters.js rotation.js js/
mv 640360barkane.jpg images/
```

После этого статика стала загружаться корректно, интерфейс отображается полностью.

## 5. Реализация импорта событий через JSON

### 5.1. Создание API-эндпоинта

В ТЗ была задача загрузки событий из JSON. Я создал файл `app/public/import.php` (впоследствии переместил в `app/` для удобства). Он принимает POST-запрос с JSON-массивом, проверяет API-ключ из заголовка `X-API-Key` и вызывает метод `EventRepository::importFromJson()`.

### 5.2. Функция PostgreSQL `import_events`

В `init-db/02-functions.sql` (я создал дополнительный файл) определил функцию:

```sql
CREATE OR REPLACE FUNCTION import_events(json_data jsonb)
RETURNS TABLE(imported integer, errors text[]) AS $$
...
```

Она проходит по каждому элементу массива, выполняет `INSERT` в таблицу `events` и возвращает количество импортированных записей и список ошибок.

### 5.3. Тестирование импорта

Первая попытка через cURL:

```bash
curl -X POST http://192.168.1.36:8099/import.php \
-H "X-API-Key: your_secret_key_here" \
-H "Content-Type: application/json" \
-d '[{"name":"Летний фестиваль","date":"2026-07-20",...}]'
```

Вернулась ошибка `Not null violation: null value in column "id"`. Анализ показал, что функция пытается вставить `id` из JSON, а его нет.

### 5.4. Исправление

Так как в таблице `id` имеет `DEFAULT nextval(...)`, я переписал функцию `import_events`, убрав колонку `id` из списка `INSERT`. Теперь `id` генерируется автоматически. Однако я решил оставить возможность передавать `id` вручную (для совместимости), поэтому оставил поле в JSON, но сделал его необязательным, используя `COALESCE(rec.id, DEFAULT)` — но проще было убрать. В итоге я оставил явную передачу `id` в JSON для управляемости.

Повторный запрос с полем `id`:

```json
{"id":1001,"name":"Летний фестиваль",...}
```

Импорт прошёл успешно, ответ `{"status":"ok","imported":2,"errors":[]}`.

## Пояснение:

**PDO (PHP Data Objects)** — это встроенное расширение PHP, предоставляющее универсальный объектно-ориентированный интерфейс для работы с различными СУБД (PostgreSQL, MySQL, SQLite и др.).

**Ключевые возможности:**

- **Подготовленные запросы** с плейсхолдерами — защита от SQL-инъекций.
- **Единый API** — код не привязан к конкретной БД, легко переключаться между драйверами.
- **Объектный подход** — работа через классы `PDO`, `PDOStatement`.
- **Обработка ошибок** через исключения.

В моём проекте именно PDO позволил безопасно заменить старый код на MySQL и обеспечить совместимость с PostgreSQL, а также использовать параметризованную фильтрацию событий.
